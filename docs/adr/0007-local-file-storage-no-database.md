# ADR-7：存储用本地文件 + 开放接口，不用数据库

## 状态
已接受

## 背景
现状追认：本决策早于本次 SDD 逆向即已成立，依据 docs/ARCHITECTURE.md §11 补记为独立 ADR。

ServerProbe 是一个"只读优先、绝不成为事故源"的运维探针，需要持久化两类数据：每次启动的结构化启动画像（StartupProfile），以及聚合后的指标历史（MetricHistory）。围绕"用什么承载这些持久化"，需要在自包含的本地文件与引入数据库之间做出选择。同时项目对外提供数据消费能力，需要决定第三方如何接入。

## 决策
存储采用本地文件（JSON/JSONL）落盘，并辅以开放接口（只读读取 API + 存储 SPI），不内置、不依赖任何数据库。

## 理由
- 探针定位为轻量、自包含、零部署依赖的组件；引入数据库会增加故障面与运维成本，与"绝不成为事故源"的设计原则相悖。
- 本地文件方案满足持久化需求即可：启动画像每次一份 JSON，指标历史以 JSONL 行式追加，按日期/会话滚动，可配保留策略与体积上限，原子写入（临时文件 + rename）。
- 通过统一的存储 SPI 写入，默认且唯一内置实现为本地文件；既保持自包含，又为有数据库/远程后端需求的使用方预留了扩展点（详见 PRD FR8.2）。
- 对外消费由开放接口承担：① 只读读取 API（最新指标快照、指标历史、启动画像，第三方经 TabooLib 服务获取，不暴露写入/控制能力）② 存储 SPI 扩展点 ③ 导出端点（Prometheus/Web）。跨服聚合交由外部系统经开放接口完成（PRD §12）。

## 后果
- 正面：零外部依赖、零部署成本、不引入数据库故障面；数据格式透明可直接消费；落盘根对象均含 `schemaVersion`（M1=1），支持格式演进与向后兼容。
- 负面/约束：本地文件存在体积膨胀风险，须以"聚合后落盘 + 滚动 + 保留策略 + 体积上限"控制；多实例下按实例 ID 分目录存放，跨服聚合不在本插件内完成，需由外部系统经开放接口实现。
- 约束：所有持久化必须经统一存储 SPI，文件写入异步且原子，严禁主线程阻塞磁盘 IO。

## 备选方案
- **EasyQuery / 数据库落库**：被否决。引入数据库会增加故障面与运维成本，违背探针轻量自包含、不成为事故源的定位。需数据库/远程后端的场景可由第三方实现存储 SPI 自行接入，本插件不内置、不依赖。
