# ADR-10：依赖策略：允许按需引入轻量依赖，逐个确认

## 状态
已接受

## 背景
现状追认：本决策早于本次 SDD 逆向即已成立，依据 docs/ARCHITECTURE.md §11 补记为独立 ADR。

ServerProbe 需在"功能"与"依赖面"之间取得平衡：完全不引入依赖会限制功能（如 Prometheus、Web 面板），无限制引入又会膨胀依赖面、增加冲突与维护成本。需要确立一条明确的依赖引入策略。

## 决策
允许按需引入轻量依赖，但每个新依赖需逐个确认。

## 理由
- 在功能与依赖面之间求平衡：既不一刀切禁止，也不无限制放开。
- 优先复用 TabooLib 自带能力，避免重复引入：JSON 优先用 TabooLib 自带 gson；Prometheus 可手写文本或按需引 simpleclient；Web 面板按需引轻量库（PRD §12）。
- 逐个确认的流程确保每个新依赖的引入理由、轻量性与必要性都经过审视。

## 后果
- 正面：依赖面可控、冲突风险低；优先复用既有能力，减少冗余。
- 负面/约束：每个新依赖引入前都需要单独评估与确认，流程上略增成本，但换取长期可维护性。
- 约束：新增依赖须优先复用 TabooLib 自带（如 gson）；确需新依赖时按"轻量 + 逐个确认"原则评估。

## 备选方案
- **完全不引入任何依赖**：被否决，会限制 Prometheus、Web 面板等功能的实现。
- **无限制引入依赖**：被否决，会膨胀依赖面、增加版本冲突与维护成本。
